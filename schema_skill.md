# Skill: /schema_analysis

## Trigger
사용자가 `/schema_analysis [PROJECT_NAME]` 을 입력하면 이 스킬이 발동된다.

---

## Behavior

사용자가 제공한 코드, 설명, README, 함수, 플러그인 등 **어떤 형태의 프로젝트 정보**든 분석하여
아래 **Ontology Schema 포맷**으로 변환해 마크다운 코드블록 안에 출력한다.

---

## Output Format (엄격히 준수)

```
# Ontology Schema: {PROJECT_NAME}

> {시스템의 목적을 한 줄로}

---

## Overview
{전체 시스템이 무엇을 하는지 2~3문장. 입력→처리→출력 흐름 중심으로}

---

## Logic Map

-> Logic A : {로직 그룹명}

{이 로직 블록의 역할과 존재 이유를 2~3문장으로 설명}

* action : a-1
   * name        : {액션명 — 동사형으로, ex. parseCSV / validateToken}
   * description : {이 액션이 하는 일 한 줄}
   * input       : "{타입}" ( ex : {실제 예시값} )
   * output      : "{타입}" ( ex : {실제 예시값} )
   * side_effects : {none | DB write | API call | event emit 등}

* action : a-2
   * name        : ...
   * description : ...
   * input       : "{타입}" ( ex : ... )
   * output      : "{타입}" ( ex : ... )
   * side_effects : ...

---

-> Logic B : {다음 로직 그룹명}

{설명}

* action : b-1
   ...

(모든 논리적 경계를 빠짐없이 Logic으로 분리할 것)

---

## Entity Relationships

* {Entity A} → {관계 동사} → {Entity B}
* {Entity B} → {관계 동사} → {Entity C}
...

* {예외 상황 또는 제약 조건}
* {예외 상황 또는 제약 조건}

---

*Generated by /schema_analysis*
```

---

## Rules (반드시 준수)

1. **Logic은 실제 경계로 나눈다** — 인증 / 파싱 / 변환 / 저장 / 알림 등 책임이 다른 곳에서 나눌 것. 억지로 묶거나 쪼개지 않는다.
2. **Action은 원자적(atomic)이어야 한다** — 하나의 action = 하나의 일.
3. **타입은 구체적으로** — `string`, `number`, `boolean`, `object`, `array<string>`, `Date`, `UUID`, `URL`, `File`, `Buffer` 등. `any` 금지.
4. **예시값은 도메인에 맞게** — 사용자 이름이면 `"kim_junho"`, 날짜면 `"2024-03-15"` 처럼 실제 같은 값.
5. **side_effects는 정직하게** — 없으면 `none`, 있으면 구체적으로 (`PostgreSQL upsert`, `Slack Webhook POST` 등).
6. **출력은 마크다운 코드블록 안에** — 그대로 `.md`로 저장 가능해야 한다.
7. **언어** — 사용자가 한국어로 작성했으면 한국어로, 영어면 영어로 출력.

---

## 사용 예시

**입력:**
```
/schema_analysis GitHub_OAuth

GitHub OAuth2 로그인 플러그인.
사용자가 GitHub 버튼 클릭 → redirect_uri + state 생성 → GitHub 인증 페이지로 redirect.
콜백에서 code 수신 → access_token 교환 → /user API로 프로필 조회 → DB upsert → JWT 발급.
```

**출력 예시 (일부):**
```
-> Logic A : 인증 요청 생성

사용자의 로그인 시도를 받아 GitHub OAuth 인증 URL을 생성하고 리디렉션한다.
CSRF 방지를 위해 state 값을 세션에 저장한다.

* action : a-1
   * name        : generateState
   * description : CSRF 방지용 랜덤 state 토큰 생성
   * input       : "void"
   * output      : "string" ( ex : "a3f9c2b1d7e4" )
   * side_effects : session write

* action : a-2
   * name        : buildAuthURL
   * description : client_id, redirect_uri, state, scope를 조합해 GitHub 인증 URL 생성
   * input       : "object" ( ex : {client_id:"abc", redirect_uri:"https://app.com/callback", state:"a3f9c2b1"} )
   * output      : "URL" ( ex : "https://github.com/login/oauth/authorize?..." )
   * side_effects : none
```
